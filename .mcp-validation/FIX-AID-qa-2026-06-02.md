## BUG-B (kyo-confirm elicitation rejected by host) ; ROOT CAUSE PINPOINTED

### Source pointers

- Demo call site: `kyo-mcp/jvm/src/test/scala/demo/Confirm.scala:33`
  ```
  requestedSchema = Json.JsonSchema.from[Unit]
  ```
- Schema test that documents the wire shape this produces:
  `kyo-schema/shared/src/test/scala/kyo/JsonTest.scala:1063`
  ```
  assert(Json.jsonSchema[Unit] == Json.JsonSchema.Obj(List.empty, List.empty))
  ```
- Serializer: `kyo-schema/shared/src/main/scala/kyo/Json.scala:229-247`
  ```
  case obj: Obj =>
      discard(entries.addOne("type" -> (() => writer.string("object"))))
      if obj.properties.nonEmpty then
          discard(entries.addOne("properties" -> ...))
      end if
      if obj.required.nonEmpty then ...
  ```

### What the wire actually looks like

`Json.JsonSchema.from[Unit]` serializes to `{ "type": "object" }` — the
`properties` field is OMITTED when empty (line 231 guard). Claude Code's
MCP elicitation validator (Zod-based, per the error report we observed)
requires `params.requestedSchema.properties` to exist as a record.

### Smallest demo-side fix (recommended)

In `Confirm.scala`:

```scala
case class ConfirmYes(yes: Boolean) derives Schema, CanEqual
...
val request = McpServer.ElicitationRequest(
    message = s"Really delete '${req.target}'? Type yes / no.",
    requestedSchema = Json.JsonSchema.from[ConfirmYes]
)
```

This emits `{ "type": "object", "properties": { "yes": { "type": "boolean" } }, "required": ["yes"] }`,
which Claude Code accepts.

### Larger fix (kyo-schema-wide, OPTIONAL)

Drop the `obj.properties.nonEmpty` guard at `Json.scala:231` so an empty
schema serializes to `{ "type": "object", "properties": {} }`. This is
the JSON-Schema-compliant shape and benefits any future caller that
genuinely wants to elicit a `Unit`-shaped object. Note this changes
the wire output for ALL empty-property objects across kyo-schema — run
existing serialization tests to confirm nothing regresses.

### Repro

`mcp__kyo-confirm__destructive_op({"target": "anything"})` → -32603 with
Zod report citing `requestedSchema.properties: expected record, received undefined`.

## BUG-A (kyo-summarize sampling rejected) ; UNDERLYING WIRE NEEDED

### Source pointer

- Demo: `kyo-mcp/jvm/src/test/scala/demo/Summarize.scala:21-52` — constructs
  `McpServer.SamplingRequest(messages, maxTokens)` and calls
  `Mcp.server.map(_.requestSampling(request))`.
- Error wrapper: line 42-44 — catches `McpException` and maps to
  `SamplingRejected(reason = ex.getMessage)`.

### Why the QA tester saw only "sampling-rejected"

The demo declares the error mapping:
```
.error[SamplingRejected](code = -32050, message = "sampling-rejected")
```
The kyo-mcp engine puts `SamplingRejected.reason` into the JSON-RPC
error response's `data` field (per the typed-error wire triple, commit
`c31c1a99c`). Claude Code's tool surface displays only the static
`message` field, not the dynamic `data`. So even though the reason is
on the wire, the QA tester can't see it.

### Diagnostic next step

Two paths, in order of cheapness:

1. **Local stderr capture**: in `Summarize.scala:42-44`, replace the
   silent recover with:
   ```
   .handle(Abort.recover[McpException] { ex =>
       Sync.defer {
           java.lang.System.err.println(s"[Summarize] requestSampling failed: ${ex.getMessage}")
           Abort.fail(SamplingRejected(reason = ex.getMessage))
       }
   })
   ```
   `Command(...).inheritStderr` is already set by the harness, but the
   proxy reroutes child stderr to `proxy-child.demo.Summarize.log` —
   tail that file after the next failing call to see the actual
   message Claude Code sent back.

2. **Wire-tee**: re-introduce the proxy.py `tee` that earlier commits
   removed (per git log: "Drop stdin tee too; same buffering problem
   in the other direction"). The earlier tee was line-buffered and
   broke the handshake. A non-blocking, append-only tee on the
   child→upstream direction only (frames matching method=`sampling/...`
   or having `id` set by us) would side-step the buffering issue.

### Expected outcomes after path 1

- If the host returns "method not allowed" → Claude Code does not
  support sampling/createMessage at all (capability gate). Then the
  demo's only sensible action is to detect the absent capability up-
  front (already advertised in `initialize` → `clientCapabilities.sampling`)
  and skip the request with a clearer error.
- If the host returns a schema-validation error → analogous to BUG-B;
  there's a shape mismatch between what kyo-mcp sends and what the host
  validates against. Fix would be in kyo-mcp's outgoing
  `SamplingRequest` serializer.

## BUG-C (kyo-http-fetch scheme downgrade) ; UPSTREAM HARDENING

### Source pointers

- Demo: `kyo-mcp/jvm/src/test/scala/demo/HttpFetch.scala:35-43`
  ```
  HttpClient
      .withConfig(_.timeout(10.seconds).followRedirects(true)) {
          HttpClient.getText(req.url)
      }
  ```
- `HttpClient.getText` signature: `kyo-http/shared/src/main/scala/kyo/HttpClient.scala:312`
  ```
  def getText(url: String | HttpUrl, ...)
      : String < (Async & Abort[HttpException]) =
      resolveUrl(url).map(u => sendUrlBody(u, routeGetText, ...))
  ```
- The JVM-side `resolveUrl` / `sendUrlBody` chain: `kyo-http/jvm/...` —
  no scheme grep hit, so the JVM `HttpClient` impl currently accepts the
  string without rejecting non-http/https schemes; the underlying
  `java.net.URI` extraction pulls host+path and the request defaults to
  HTTP for the host.

### Smallest demo-side fix

In `HttpFetch.scala:31-34`, before calling `HttpClient.getText(req.url)`:

```scala
val parsed = scala.util.Try(java.net.URI.create(req.url)).toOption
val isHttp = parsed.exists(u => u.getScheme == "http" || u.getScheme == "https")
if !isHttp then
    Abort.fail(HttpError(s"unsupported scheme: ${parsed.map(_.getScheme).getOrElse("<unparseable>")}"))
else ...
```

### Better fix (kyo-http)

Add scheme validation to `kyo-http/shared/src/main/scala/kyo/HttpClient.scala`
`resolveUrl(...)` or to the URL-resolution pipeline so any non-http(s)
scheme aborts with a typed `HttpException`. Benefits all downstream
consumers and closes the SSRF surface uniformly.

### Repro

`mcp__kyo-http-fetch__http_get({"url": "ftp://example.com/"})` returns
the example.com HTML body. Should return `-32020 http-error`.

## BUG-D (kyo-filesystem search follows symlinks) ; UNDERLYING DATA EXFIL

### Source pointer

`kyo-mcp/jvm/src/test/scala/demo/Filesystem.scala:107-128`

```
val searchTool =
    McpHandler.tool[SearchFiles](...) { req =>
        val rootParts = root.parts
        root.walk
            .filter(_.isRegularFile)
            .collect { file =>
                file.read
                    .handle(Abort.recover[FileReadException](_ => ""))
                    .map { contents =>
                        if contents.contains(req.pattern) then Present(...)
                        else Absent
                    }
            }
            .run
            .map(hits => McpContent.text(hits.mkString("\n")))
            .handle(Scope.run)
    }
```

### Why it's broken

`Path.walk` returns symlinks as well as regular files. `file.read` then
opens the symlink's target. If the target is outside `root` (e.g.
`/etc/passwd` reached via `/tmp/mcp-validation/root/symlink-escape ->
/etc/passwd`), `file.read` succeeds and the content gets searched —
leaking adjacent bytes from the system file via a narrow-band pattern
oracle.

### Smallest fix (uses already-shipped API)

```
val rootReal = root.realPath  // pre-resolve once
root.walk
    .filter(_.isRegularFile)
    .collect { file =>
        file.confinedTo(root)
            .handle(Abort.recover[FileException](_ => null))
            .map { confined =>
                if confined eq null then Absent
                else confined.read.handle(Abort.recover[FileReadException](_ => "")).map { contents =>
                    if contents.contains(req.pattern) then Present(...)
                    else Absent
                }
            }
    }
```

The `Path.confinedTo` introduced in commit `3e6847025` IS the right tool;
it just was not applied to the search path. `read_file` already uses an
equivalent guard, hence the inconsistency that surfaced this bug.

### Repro

Setup:
```
ln -sf /etc/passwd /tmp/mcp-validation/root/symlink-escape
```
Then `mcp__kyo-filesystem__search_files({"pattern":"_uucp"})` returns
`symlink-escape`, because the search content-read followed the symlink to
`/etc/passwd` which contains `_uucp:*:4:...`.

## BUG-E (kyo-http-fetch maxBytes ≤0) ; INPUT VALIDATION

### Source pointer

`kyo-mcp/jvm/src/test/scala/demo/HttpFetch.scala:39-43`

```
.map { body =>
    val out = if body.length > cap then body.take(cap) + s"\n…[truncated at $cap bytes]" else body
    McpContent.text(out)
}
```

where `cap = req.maxBytes.getOrElse(1024 * 1024)`.

### Why it's broken

With `cap = 0`, `body.length > 0` is true (body is non-empty for a 200
response), so the branch returns `body.take(0) + "\n…[truncated at 0 bytes]"`,
which is the truncation message but with no body. With `cap = -100`,
`body.take(-100)` returns empty string in Scala, same outcome.

### Smallest fix

Clamp `cap` to a minimum of 1 (or document the behavior). Or reject
negative values:
```
val cap = req.maxBytes.getOrElse(1024 * 1024)
if cap < 1 then Abort.fail(HttpError(s"maxBytes must be >= 1, got $cap"))
else ...
```

## QUIRK-C / QUIRK-E (single-code error families) ; TYPED ERROR SPLIT

### Pattern

Both `kyo-filesystem` (-32001 filesystem-error) and `kyo-http-fetch`
(-32020 http-error) collapse multiple distinguishable failure modes into
a single error code. The fix is the same shape: split the typed error
ADT and assign distinct codes.

### Recommended typed split (filesystem)

```
sealed trait FsError derives Schema, CanEqual
object FsError:
    case class NotFound(path: String) extends FsError       // -32001
    case class AccessDenied(path: String) extends FsError   // -32002
    case class NotARegularFile(path: String) extends FsError // -32003
    case class OutsideRoot(path: String) extends FsError    // -32004 (BUG-D)
    case class IoError(reason: String) extends FsError      // -32005
```

Then in the handler chain, map each underlying exception
(`FileNotFoundException`, `FileAccessDeniedException`,
`FileReadException`, etc.) to the corresponding variant.

### Recommended typed split (http-fetch)

```
sealed trait HttpError derives Schema, CanEqual
object HttpError:
    case class BadScheme(scheme: String) extends HttpError    // -32020
    case class DnsFailure(host: String) extends HttpError     // -32021
    case class ConnectionRefused(target: String) extends HttpError // -32022
    case class Timeout(after: Duration) extends HttpError     // -32023
    case class StatusError(code: Int, body: String) extends HttpError // -32024
    case class TooLarge(limit: Int) extends HttpError         // -32025
```

The `kyo-mcp` registered-resource example (`-32002 Unknown resource ...
Registered: ...`) is the model: typed code + message + helpful data.

## BUG-I (proxy.py: no auto-respawn on JVM-child EOF)

### Source pointer

`.mcp-validation/proxy.py:154-159`

```
def child_to_upstream(c: subprocess.Popen):
    while not state["shutdown"]:
        body = read_one_message(c.stdout)
        if body is None:
            log(f"child {c.pid} stdout EOF")
            return                              # <-- just exits; no respawn
```

### Smallest fix

Replace `return` with a recovery call:

```
if body is None:
    log(f"child {c.pid} stdout EOF — auto-respawning")
    if not state["shutdown"]:
        trigger_reload()
    return
```

`trigger_reload()` (already defined at line 210) kills any still-alive
child and spawns a new one. The recovery would now fire on either
sentinel-touch OR child-death. Safe to call from the reader thread
because `trigger_reload` already serializes via `state["lock"]`.

## BUG-J (proxy.py: sbt-lock contention on simultaneous reload)

### Source pointer

`.mcp-validation/proxy.py:210-251` (`trigger_reload` body — runs
`sbt <module>/Test/compile` synchronously).

### Why it's broken

10 proxies × 1 sentinel touch = 10 `sbt compile` invocations against the
same `~/.ivy2/.sbt.ivy.lock`. Sbt fails silently in 4-of-10 cases under
contention; the proxy logs `reload compile failed` and stops without
retry. Net effect: half the servers stuck after one sentinel touch.

### Smallest fix (proxy-local)

Add a retry on compile failure:

```
def trigger_reload():
    log("reload triggered")
    for attempt in range(3):
        try:
            subprocess.run([LAUNCHER, ...], check=True, ...)
            break
        except subprocess.CalledProcessError as e:
            log(f"reload compile failed (attempt {attempt+1}): {e}")
            if attempt == 2:
                log("giving up after 3 attempts")
                return
            time.sleep(2 + attempt * 3)   # back off
    ...
```

### Better fix (cross-proxy serialization)

A single flock-guarded compile would prevent contention entirely:

```
with open("/tmp/mcp-validation/sbt-compile.lock", "w") as f:
    fcntl.flock(f, fcntl.LOCK_EX)
    subprocess.run([LAUNCHER, ...])
```

## BUG-K (Claude Code LSP plugin: didOpen not re-sent after server restart)

### Where to file

Upstream to Claude Code (https://github.com/anthropics/claude-code/issues).
This is in the LSP plugin client, not in kyo-lsp / kyo-jsonrpc — verified
by: a FRESH file (which the plugin client has never opened) works
correctly against the new JVM, while a previously-opened file returns
"No hover information".

### Repro

1. Open file `/tmp/foo.todo` via `LSP hover` — works.
2. `touch .mcp-validation/reload.sentinel` (kills + respawns the LSP JVM).
3. Re-call `LSP hover` on `/tmp/foo.todo` — returns "No hover
   information". Same query on a file the client has never opened before
   — works correctly.

### Workaround

Restart Claude Code, OR open the file in a fresh editor session.

## Cross-cutting recommendations

1. **Pinpoint reverse-direction wire shape**. Before fixing BUG-A,
   capture one full `sampling/createMessage` outgoing JSON and the
   matching response so the fix-implementer has the exact spec gap.
   Easiest path: BUG-A diagnostic step 1 above (add a stderr log line in

   sampling method.
2. **Run the headless harness after every demo-side fix**. It already
   covers the 5 in-tree demos end-to-end and would catch a regression
   instantly.
3. **Add a smoke test for each typed-error code** the demos surface, so
   QUIRK-C/QUIRK-E splits don't regress.
4. **Document the `notifications/cancelled`, `resources/subscribe`,
   `completion`, `executeCommand` Claude-Code-host gap** in the
   demos' READMEs. The headless harness is the validation path for
   those features; spelling that out keeps future QA pass time bounded.
