Run this as an opus agent once IMPLEMENTATION.md exists. The agent VALIDATES the plan against the /implement skill's 9 plan-as-contract rules. Output is PASS or a list of specific failures to be fixed by a follow-up sonnet agent.

```
Validate `kyo-jsonrpc/IMPLEMENTATION.md` against the /implement skill's plan-as-contract rules.

Working directory: /Users/fwbrasil/workspace/kyo/.claude/worktrees/crispy-swinging-lemur

## Inputs
- kyo-jsonrpc/IMPLEMENTATION.md (the plan under validation)
- kyo-jsonrpc/DESIGN.md (the spec the plan must realize)
- kyo-jsonrpc/STEERING.md (kyo conventions)
- kyo-jsonrpc/audit/round2/COMPLETENESS.md, CORRECTNESS.md, CONSISTENCY.md, CLARITY.md
- kyo-jsonrpc-PROMPT.md (the original prompt; for test category cross-check)

## Rules (each must PASS for validation to succeed)

1. **Every open input item appears in the plan.** Every "Critical" and "Important" finding from round-2 audits maps to a phase. None silently dropped, none folded into "future work", none labeled "documented".

2. **No priority/preference language.** Grep the plan for "priority", "tier", "first", "most important", "if time permits", "low priority", "nice to have", "edge case as cut". Each occurrence is a violation.

3. **Concrete content per phase.** Each phase must list, by exact path: Files to produce (full path + one-line purpose), Files to modify (full path + specific edit), Files to delete (if any), Tests (numbered leaves, one sentence each, total count stated), Public API additions/modifications/removals (signature-level), Verification command (exact shell invocation).

4. **No vague phrasings.** Banned: "we'll figure it out", "tighten as we go", "investigate further", "reconsider after Phase X", "TBD", "to be decided", "depending on", "polish". Every cell must be concrete.

5. **Deviations require explicit user accept.** No phase silently omits a DESIGN.md element. If the plan deviates from DESIGN.md, the deviation is called out in the phase's notes with rationale.

6. **Test scenarios are listed, not summarized.** "Concurrency tests" is NOT a scenario; "two concurrent acquires on a permit-1 meter; the second blocks until the first releases" IS. Every leaf must be single-sentence specific.

7. **Phase ordering is dependency-justified.** For each phase N>0, the plan states which earlier phase(s) it depends on and why (specific API/type/file). No hand-waved dependencies.

8. **Zero open items in the summary/plan.** No "open items the user should decide", "options to choose between", "TBD", "future decision", "to be confirmed at impl time", "impl agent will decide X".

9. **Implicit-open-item hunt.** Look for: deferred decisions, undocumented assumptions, missing platform paths, vague test verifications, missing error-path tests, missing anti-flakiness measures, forward-reference compile-isolation violations. Each finding listed.

## Additional kyo-jsonrpc-specific checks

10. **All 11 prompt-required test categories appear** (response with both result+error rejection, Maybe[JsonRpcId] Absent→no-key, notification-failure-no-response, notification-no-wire-reply, transport-close-mid-call, bidi A↔B concurrency, cancel-arrives-before-reply, cancel-arrives-after-reply, progress 3-emission, partial-result accumulator, scope-cleanup-fails-pending). Cite the phase + test number for each.

11. **All round-2 critical findings have a concrete test.** C1 (engine-emitted extras), C3 (CAS race close), C4 (callerRegistry timing), C6 (no derives Schema on Envelope) — each must appear as a test leaf with single-sentence specificity.

12. **Cross-platform commitment is honored.** Tests live in `shared/`. Each test leaf MUST work on JVM, JS, and Native. Any leaf that would not work cross-platform must be marked with explicit JVM-only justification (none expected).

13. **The Exchange type-parameter pinning is consistent across phases.** Phase 4 declares Exchange's type params; Phase 5 (cancellation) does not contradict.
```
