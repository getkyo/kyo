## Summary of root causes (3 distinct bugs)

**Bug A (7 tests affected: 1, 2, 4, 7, 10-ish, 13, 14):** `ProgressEngine.buildProgressSink` guards sends on `pendingInbound.get(id) == InboundEntry.Running`. The handler fiber calls `ctx.progress` early, but `pendingInbound.put(id, entry)` occurs AFTER `Fiber.initUnscoped` returns in the decode callback's `Sync.Unsafe.defer` block. If the fiber runs before the defer completes (scheduler can interleave), the guard sees `null`/`Absent` and drops the notification silently. Fix: call `pendingInbound.put` BEFORE `Fiber.initUnscoped`, or remove the running-guard from `buildProgressSink` (the token already scopes the notification correctly).

**Bug B (2 tests affected: 3, 12):** The progress channel stores the full notification params `{token, value}` but `callPartialResults` tries to decode the stored value as `T` directly. The `value` field must be extracted before storing in the channel (or `callPartialResults` must extract the field when consuming). LSP uses `"value"` key; MCP embeds it differently. Add a `extractProgressValue: Structure.Value => Structure.Value` field to `ProgressPolicy` (or extract inline in the decode callback).

**Bug C (2 tests affected: 5, 6):** `subscribeProgress` registers the channel lazily inside the `Stream` body. The channel is not live until `stream.run` is called. Any notification sent between `subscribeProgress` and `stream.run` is lost. The eager-registration pattern requires either returning `Stream < Sync` or using a `Ref` that is populated at call time and read lazily in the stream body.

## Recommendation: STEER

Three distinct bugs, all straightforward to fix:

1. **Bug A (highest priority, most tests):** Move `pendingInbound.put(id, entry)` before `Fiber.initUnscoped(handlerEffect)` in `JsonRpcEndpointImpl.scala` lines 668-671. This unblocks 7 tests.
2. **Bug B:** Extract the `value` field from notification params before offering to progress channel in the decode callback (lines 619-628). Add `extractProgressValue` to `ProgressPolicy` or inline per-policy logic.
3. **Bug C:** Make `subscribeProgress` register the channel eagerly. Change signature to return `Stream[...] < Sync` and perform channel init in `Sync.defer` outside the `Stream` body, then build the stream over the pre-created channel.
