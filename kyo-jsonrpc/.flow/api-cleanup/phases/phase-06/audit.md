## NOTE findings

- **NOTE-1: Return-row widening to `Abort[Throwable]` deviates from design §7's `(Async & Scope)` signature, but the deviation is correct.** Design §7 row 1 of the cross-platform table specifies `JsonRpcTransport < (Async & Scope)`, and D4 §5 code block at line 97 uses the same row. Implementation chose `(Async & Scope & Abort[Throwable])`. The deviation is necessary: the JS/Native stubs depend on `Abort.fail` to surface the unsupported-platform error, and absorbing that into `Abort[Nothing]` (i.e. erasing it) would require either `Abort.run` (which discards the error) or a non-existent `Abort.faili` infallible variant. Decisions.md "Return type" section captures the rationale. Recommendation: when the next surface-state doc (`A2` refresh or `D4` follow-up) is written, mention the `Abort[Throwable]` row so the design corpus stays in sync with the code. Not commit-blocking.

- **NOTE-2: Test placement deviates from design §7's `shared/src/test` target, but the deviation is correct.** Design §7 paragraph 2 says the test "relocates to `kyo-jsonrpc/shared/src/test/scala/kyo/JsonRpcTransportUnixTest.scala`. On JVM it runs the full UDS flow; on JS and Native it catches the `UnsupportedOperationException` and cancels, matching `kyo-http/shared/src/test/scala/kyo/HttpServerUnixTest.scala:17`." Implementation kept it at `kyo-jsonrpc/jvm/src/test/scala/kyo/JsonRpcTransportUnixTest.scala`. Decisions.md "Test placement" justifies this on the grounds that the assertions use JVM-only NIO classes (`SocketChannel`, `UnixDomainSocketAddress`, `ByteBuffer`) as a direct client; a shared-test version would need platform-specific stub clients on JS/Native and would add complexity for no coverage gain because the JS/Native paths are tautological `Abort.fail` stubs. This is sound — the kyo-http precedent works because kyo-http has real UDS implementations on all three platforms; kyo-jsonrpc does not yet, so the shared-test pattern would be premature. The plan-level note quoted in decisions.md ("simpler: keep the test JVM-only by leaving it in kyo-jsonrpc/jvm/src/test/") indicates this deviation was anticipated during planning. Recommendation: when Native/JS real UDS implementations land (follow-up task), move the test to `shared/src/test` as design §7 originally specified, and add JS/Native specialisations.

- **NOTE-3: Stale references to `JsonRpcTransportJvm` exist in non-source artefacts.** `kyo-jsonrpc/dev-strip-protocol-coverage-jsonrpc.json:4` references the now-deleted path `kyo-jsonrpc/jvm/src/main/scala/kyo/JsonRpcTransportJvm.scala` as part of an `allow` list. This file is from a prior campaign (Rule 8 cleanup, commit `251602de6`) and is not consulted at build time. It is also not on the post-strip cleanup path (the file is committed; the campaign that produced it is closed). Two test-report XMLs under `kyo-jsonrpc/jvm/target/test-reports/TEST-kyo.JsonRpcTransportJvmTest.xml` and a `succeeded_tests` cache also reference the old test class name; both are sbt-managed build artefacts that will rebuild on next `sbt clean test` or `sbt test`. The historical-state and decision docs under `kyo-jsonrpc/.flow/api-cleanup/` (`A2`, `C-cleanup-plan`, `D4`) legitimately mention the deleted name; those should NOT be rewritten — they are the audit trail for why the deletion happened. No action required.
