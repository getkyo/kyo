# Research finding Q-002

Question: Which ZLIB inflate strategy should the JS and Native InflateHook implementations adopt for cross-platform RFC 1950 parity with the JVM InflaterInputStream?
Type: research-knowable
Budget: 10 minutes (used: 10)

## Answer

Scala Native already ships `java.util.zip.InflaterInputStream` as part of its javalib, backed by `scala.scalanative.ffi.zlib` (libz FFI). The Native `InflateHook` can therefore mirror the JVM implementation verbatim: wrap `ByteArrayInputStream` in `InflaterInputStream`, drain to `ByteArrayOutputStream`, close, return the bytes. No pure-Scala inflate is needed on Native. For Scala.js, `java.util.zip` is absent from the Scala.js javalib entirely, and no maintained pure-Scala inflate/deflate library with JVM+JS+Native artifact parity exists on Scaladex or GitHub as of May 2026; the closest contender (fs2 compression) uses Node.js `zlib` module interop on JS, not pure Scala. The correct strategy is therefore: (1) promote the Native `InflateHook` to use `java.util.zip.InflaterInputStream` identically to JVM, and (2) implement an in-tree pure-Scala RFC 1950 inflate in `shared/` that the JS `InflateHook` delegates to, keeping the same `inflate(Array[Byte]): Array[Byte] < (Sync & Abort[TastyError])` contract as the JVM hook. This avoids an external dependency on an unmaintained or JS-only library, matches the in-tree posture already established for cross-platform parity, and eliminates the current `NotImplemented` failure on both JS and Native.
