# Phase 25b Decisions

## Utf8 encode API choice

`Utf8` exposes only `decode(bytes: Array[Byte], offset: Int, length: Int): String`. There is no
`encode` method. The encode side of the round-trip uses `s.getBytes(StandardCharsets.UTF_8)`, which
invokes the same platform UTF-8 codec that each `Utf8` platform implementation wraps (JVM and Native:
`StandardCharsets.UTF_8`; JS: `TextDecoder`, whose counterpart encoder is `TextEncoder` which
`getBytes` delegates to via Scala.js internals). The round-trip therefore exercises `Utf8.decode`
against well-formed UTF-8 bytes produced by the platform's own encoder, confirming consistency.

## Supplementary character generation strategy

Supplementary code points (U+10000..U+10FFFF) are generated by calling `Character.toChars(cp)` and
appending the resulting `Array[Char]` as a `new String(...)`. This produces a well-formed UTF-16
surrogate pair in the Java `String` internal representation, avoiding the `getBytes` silent-lossy-
encode problem that arises from using raw surrogate `char` values. 10% probability per position is
applied (`rng.nextInt(10) == 0`). BMP code points are drawn from U+0020..U+D7FF, which excludes
the surrogate range (U+D800..U+DFFF) and ASCII control codes.

## Seed used

All three generative tests use `seed = 0L` via `new scala.util.Random(0L)`.
