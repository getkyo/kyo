# Contributing to kyo-aeron

Module-specific guide for kyo-aeron. Read the repository-root [CONTRIBUTING.md](../CONTRIBUTING.md) first: it carries the conventions, naming rules, type vocabulary, test patterns, and the unsafe-boundary tiers that apply across all of Kyo. This document records only what is specific to kyo-aeron: its uniform public API, its two-transport architecture, the wire contract, the unsafe-tier boundary, the eager-in-band connect model, the no-blocking discipline in both Scala and C, the C shim memory model, the libaeron build and link, and its cross-platform test convention.

## What kyo-aeron is

kyo-aeron is typed publish-subscribe messaging over [Aeron](https://aeron.io/): low-latency inter-process communication through shared memory (`aeron:ipc`) and UDP for remote services (`aeron:udp?endpoint=...`). The public surface is:

- `Topic` (`run`, `publish`, `stream`, `isolate`, `defaultRetrySchedule`), an opaque alias of `Env[AeronTransport]`. `run` has three overloads (`run(v)`, `run(aeronDir)`, `run(client)`; see [The public API](#the-public-api)).
- `AeronClient` (`connect`), an opaque, `Scope`-managed handle to a connected client against an external driver, shareable across `Topic.run(client)` scopes.
- the `TopicException` hierarchy (top-level in `package kyo`: `TopicBackpressureException`, `TopicPublishException`, `TopicTransportException`, and their concrete leaves).

It runs identically on JVM, Scala Native, and Scala.js (Node), with the same public types, the same effect rows, and the same on-wire bytes on every platform.

`Topic`, `AeronClient`, and the `TopicException` hierarchy are the only public top-level types. Everything else (`AeronTransport`, `AeronRuntime`, `AeronPlatform`, the per-platform `AeronPlatformTransport` selectors, `JvmAeronTransport`, `FfiAeronTransport`, `AeronBindings`, `AeronSentinels`, the `Envelope` frame) is `private[kyo]` internal. Both `Topic` and `AeronClient` are opaque aliases of an internal form, so neither `io.aeron.*` (JVM) nor the C `aeron_t` (FFI) leaks through a public type.

## Two-transport architecture

kyo-aeron binds Aeron two different ways and keeps all Kyo logic above both.

- **JVM uses the pure-Java io.aeron client.** `JvmAeronTransport` (under `jvm/src/main`) wraps `io.aeron.Aeron`, `io.aeron.driver.MediaDriver`, `BufferClaim`, and `FragmentAssembler`. The JVM Aeron stack is pure Java plus Agrona `Unsafe` plus mmap, with no JNI, so the JVM artifact has zero native dependency, a low JDK floor, and the zero-copy `tryClaim` hot path. Routing JVM through FFI would regress the primary latency platform, so it does not.
- **Native and JS use a C shim over the Aeron C client.** `kyo_aeron.c` (under `shared/src/main/c`) wraps the Aeron C client (`aeronc.h`) and the embedded C media driver (`aeronmd.h`), statically linking libaeron 1.50.2. It is bound through kyo-ffi by the `AeronBindings` trait, and `FfiAeronTransport` (under `shared/src/main`) adapts the bindings to the shared transport contract. Native and JS share the identical shim, bindings trait, and `FfiAeronTransport`; the only per-platform file is the selector.

The seam between the two is `private[kyo] trait AeronTransport` (`shared/src/main/scala/kyo/internal/AeronTransport.scala`): a small, platform-neutral op set (publication and subscription lifecycle plus the non-blocking hot-path calls). `Topic` aliases `Env[AeronTransport]` and is the sole reference to the trait. The transport handle type for publications and subscriptions is an abstract type member each impl fixes (JVM: `io.aeron.Publication`; FFI: an `Ffi.Handle`), so neither `io.aeron.*` nor `Ffi.Handle` leaks into the neutral trait or the public surface. **No `io.aeron.*` type appears anywhere on the public surface: io.aeron is a JVM implementation detail confined to `jvm/src/main`.**

The per-platform driver/connect selectors live behind `private[kyo] AeronPlatform` (`shared/src/main`), which delegates to the per-platform `AeronPlatformTransport` object (one file per platform). `AeronPlatform.embedded(dir)` launches an embedded driver and connects a client; `AeronPlatform.external(aeronDir)` connects a client to an external driver and is the shared eager-typed connect primitive consumed by both `Topic.run(aeronDir)` and `AeronClient.connect` (see [The connect model](#the-connect-model)). Each returns an `AeronRuntime` bundling the `AeronTransport` impl with an unsafe-tier `close()` that tears down what the runtime owns (embedded: client then driver; external: client only).

When adding a transport operation, add it to the neutral `AeronTransport` trait first, implement it in both `JvmAeronTransport` and `FfiAeronTransport`, and bridge it at the `Topic` layer. Do not put any `io.aeron.*` or `Ffi.*` type on the trait or in `Topic.scala`.

## The public API

The public entry points are uniform across JVM, Native, JS, and Wasm: the same signatures, the same effect rows, the same backend selection happening behind `AeronPlatform`. They live in `shared/src/main/scala/kyo/Topic.scala` and `shared/src/main/scala/kyo/AeronClient.scala`. There are no platform-specific public overloads, no JVM-only `Topic.run(MediaDriver)` / `Topic.run(Aeron)`, and no `TopicPlatformMethods` mixin with per-platform stubs. `TopicUniformInvariantsTest` "no io.aeron on public surface ..." pins their absence with `typeCheckFailure`.

The three `Topic.run` overloads all funnel through the single private `runWith` (`Env.run(transport)`):

- **`run(v)`** runs `v` against an embedded media driver, row `A < (Async & S)`. It allocates a unique per-instance directory via `Path.tempDir` (so concurrent `Topic.run` calls never collide on the CnC file), launches the embedded driver + client through `AeronPlatform.embedded(dir)`, and brackets teardown with `Sync.ensure` (close the client and driver, then delete the dir).
- **`run(aeronDir: Path)(v)`** connects a fresh client to an external driver already running at `aeronDir`, row `A < (Async & Abort[TopicTransportFailedException] & S)`. It does not start or stop the driver (the caller owns the driver lifecycle); `Sync.ensure` closes only the client. A failed connect aborts `TopicTransportFailedException` eagerly and in-band (see below).
- **`run(client: AeronClient)(v)`** borrows a caller-owned `Scope`-managed client, row `A < (Async & S)`. It does NOT close the client (the `Scope` that produced it owns its lifetime), so multiple `run(client)` scopes can share one `AeronClient` and amortize the connect cost.

`AeronClient.connect(aeronDir: Path)` is the cross-platform factory for a shared client, row `AeronClient < (Scope & Async & Abort[TopicTransportFailedException])`. It is the sole factory (the use / `initUnscoped` ladder is deliberately not multiplied); the client closes exactly once on normal, error, or cancellation scope exit via `Scope.acquireRelease`. `aeronDir` is a `kyo.Path` on every backend, so the surface is identical platform to platform.

## The wire contract

The on-wire format is a property of the module, not of any one platform, and it lives in exactly one place: shared `Topic.scala`.

- **Stream id is `tag.hash.abs`.** One Aeron stream per exact Scala type. A `stream[Parent]` cannot receive messages a publisher sent as a subtype, because the stream ids differ.
- **The frame is an `Envelope`.** Messages are `private[kyo] case class Envelope[A](typeTag: String, messages: Chunk[A]) derives Schema`, encoded with the kyo-schema MsgPack codec (`MsgPack.encode(Envelope(tag.show, messages))` produces a `Span[Byte]`; `MsgPack.decode[Envelope[A]](Span.from(bytes))` reverses it). The envelope makes the top-level encoded value always a message, so bare-primitive streams (`stream[Int]`, `stream[String]`) round-trip as the `messages` repeated field. The receiver verifies `envelope.typeTag == tag.show` before emitting, which guards against a `tag.hash` collision routing the wrong type onto a stream id.
- **The codec is kyo-schema MsgPack.** `publish[A: Schema]` and `stream[A: Schema]` take a `Schema[A]` directly. There is no upickle dependency and no `AsMessage` alias.
- **Any `Schema`-derivable type round-trips.** The only requirement is a `Schema[A]`; `MsgPack` encodes the open generic `Envelope[A]` and default-valued fields directly, with no field-shape restriction on the message type.

The wire bytes are byte-identical across JVM, Native, JS, and Wasm. A Java publisher and a C subscriber (or the reverse) interoperate over the same wire, which is why the format and the offer-sentinel error policy live only in shared `Topic.scala`, never in a transport impl. Changing the envelope shape, the stream-id derivation, the codec, or the sentinel mapping is a breaking change to the wire. Do it deliberately, across both impls, with the cross-platform byte-identity tests updated.

### Offer sentinels

`AeronTransport.offer` returns the raw Aeron position (`> 0`) or a negative sentinel. The mapping from sentinel to typed `TopicException` leaves lives once, in the shared `AeronSentinels` object and the `Topic.mapOfferResult` method. The six values (`NotConnected = -1`, `BackPressured = -2`, `AdminAction = -3`, `Closed = -4`, `MaxPositionExceeded = -5`, `Error = -6`) are byte-identical to the JVM `io.aeron.Publication` constants and to the Aeron C return codes, so one shared match serves both transports. Keep new error handling on the raw `Long` in `Topic.mapOfferResult`; impls return the raw value and never raise a Kyo error. `mapOfferResult` has no `case _` arm: Aeron's contract fixes the return-value set, so an unmatched value is a defect (a `MatchError`), not a transport condition to swallow.

An oversize message is caught before the offer, not by a sentinel. `Topic.publish` reads `AeronTransport.maxMessageLength(publication)` (FFI: `kyo_aeron_publication_max_message_length` via `aeron_publication_constants`; JVM: `Publication.maxMessageLength()`) and aborts with `TopicMessageTooLargeException(size, max)` when the encoded batch exceeds it, identically on every platform. The `-6` (`Error`) sentinel arm and the JVM `IllegalArgumentException` catch are belt-and-suspenders for the rare race where the up-front check is bypassed; both normalize to the same `TopicMessageTooLargeException`.

A `MsgPack.decode` `Result.Failure` and a `typeTag` mismatch both map through Kyo error handling (`Abort.panic`), never a thrown exception leaking out of `Topic`.

## The error model

The public failure surface is the `TopicException` hierarchy (top-level in `package kyo`, one file `TopicException.scala`), not kyo-core `Closed` or a nested `Topic.Backpressured`. It mirrors `kyo-http`'s `HttpException`: a `sealed abstract class TopicException(message, cause)(using Frame) extends KyoException`, three `sealed abstract` subcategories by failure mode, and a concrete `case class` leaf per condition with its message inline in the `extends ...(s"...")` clause. The three subcategories split retryable from terminal:

- **`TopicBackpressureException`** is the single retryable subcategory: transient flow-control, not-yet-connected, and admin-action conditions. The shared `Retry[TopicBackpressureException](retrySchedule)` loop in `Topic.scala` absorbs them; they surface terminally only as `TopicBackpressureExhaustedException` after the schedule exhausts. The sentinel arms that route here are `BackPressured` (`-2`), `NotConnected` (`-1`, consistent with the not-connected pre-check), and `AdminAction` (`-3`, a log-rotation / term-count race Aeron documents as retryable).
- **`TopicPublishException`** is terminal on the offer path of an established publication: `TopicPublicationClosedException` (sentinel `-4` or a closed client), `TopicMaxPositionExceededException` (`-5`), `TopicMessageTooLargeException` (the oversize check, or sentinel `-6`). Reachable on `publish` only.
- **`TopicTransportException`** is terminal on the add / lifecycle path: `TopicRegistrationFailedException` (the driver rejected a registration, carrying the driver errcode and detail), `TopicAddTimeoutException` (the bounded Scala add loop expired), `TopicTransportFailedException` (a fatal conductor error surfaced through the non-exiting handler, and the uniform connect-failure leaf; see [The connect model](#the-connect-model)). Reachable on both `publish` and `stream`. `TopicRegistrationFailedException.errorCode` is the raw POSITIVE driver code on every platform: the JVM reads `RegistrationException.errorCodeValue()`; the FFI path takes `math.abs(aeron_errcode())` at both capture sites (`FfiAeronTransport.pollAddPublication` / `pollAddSubscription`), because `aeron_errcode()` reports `AERON_CLIENT_ERRORED_MEDIA_DRIVER` as the NEGATED code. So the same driver rejection (e.g. a malformed `aeron:invalid` URI gives `errorCode == 1`, INVALID_CHANNEL) renders an identical positive code on JVM, JS, Native, and Wasm; `TopicUniformInvariantsTest` "TopicRegistrationFailedException.errorCode is positive ..." pins it. `math.abs` is applied only at those two capture sites: no row widens and no message string is added there (the meaning lives once on the leaf in `TopicException.scala`).

The effect rows follow the split exactly: `publish` aborts with `Abort[TopicBackpressureException | TopicPublishException | TopicTransportException]`; `stream` with `Abort[TopicBackpressureException | TopicTransportException]` (the offer-path failures are unreachable on the subscribe side, so `stream` does not carry `TopicPublishException`).

Every human-readable failure message lives only in `TopicException.scala`, in the leaf's `extends ...(s"...")` clause. `Topic.scala` and the transport impls construct typed leaves with no inline message strings; a leaf that needs field normalization (e.g. `TopicRegistrationFailedException`'s empty-detail placeholder) uses a `private` constructor plus a companion `apply`, never a message at the construction site. When adding a failure mode, add a leaf under the right subcategory in `TopicException.scala` and route to it; do not introduce a message string anywhere else, and do not widen an effect row to a broader exception type than the subcategory the condition belongs to.

## The unsafe-tier boundary

`AeronTransport`, `AeronRuntime`, `AeronClient.Unsafe`, and `AeronBindings` are unsafe-tier: every effecting method carries a trailing `(using AllowUnsafe)`, returns plain values (no effect row), and never re-enters the effect system. The exception is the two `@Ffi.blocking` binding methods (`driverStart`, `clientConnect`), whose generated signature returns `Fiber.Unsafe[..., Any]`; consumers bridge them via `.safe.get` into `Async`. `Topic`, `AeronClient`, and the `AeronPlatformTransport` selectors bridge all of these to their `< (... & Async)` rows.

The bridge is `Sync.Unsafe.defer`, never `import AllowUnsafe.embrace.danger`. `Sync.Unsafe.defer` is a context function that already supplies the `AllowUnsafe` proof to its body; the proof then threads from the call site down through each `AeronTransport.<op>(using AllowUnsafe)` to each `AeronBindings.<op>(using AllowUnsafe)`. `embrace.danger` appears nowhere in kyo-aeron main sources. There is no library-invoked callback that would need an ambient `AllowUnsafe`, because every Aeron callback stays in C (see the fragment handler below).

`AeronClient` is an opaque alias of its `AeronClient.Unsafe` form, mirroring the two-tier safe/unsafe pattern: `AeronClient.Unsafe` is a `sealed abstract class` exposing `close()(using AllowUnsafe)`, a `private[kyo] transport: AeronTransport`, and `safe: AeronClient`; `AeronClient.Unsafe.fromRuntime(runtime)` wraps an ALREADY-connected `AeronRuntime` into the unsafe handle. `AeronClient.connect` performs the connect via the shared `AeronPlatform.external` primitive (which owns the eager-in-band failure catch), then `fromRuntime` only wraps the connected runtime; it does not re-catch.

The transport selectors differ by what each backend's primitive call needs. On JVM a plain Java-client connect (`Aeron.connect`) and `MediaDriver.launchEmbedded` need no `AllowUnsafe`, so both the JVM `external` and `embedded` selectors use plain `Sync.defer` (the safe tier). The FFI selectors use `Sync.Unsafe.defer` plus `.safe.get` on the `@Ffi.blocking` `driverStart`/`clientConnect` everywhere. Either way the public rows are identical.

When adding a transport op, keep it unsafe-tier (plain value, trailing `using AllowUnsafe`), bridge it at the `Topic` layer with `Sync.Unsafe.defer`, and mark each off-heap-buffer site with a `// Unsafe:` comment. Scope `AllowUnsafe` to the narrowest method or block; never put it on a class or constructor; keep it the last `using` clause.

## The connect model

The external-driver connect is eager and in-band on every backend, and surfaces a single typed failure. Connecting to a directory where no driver is running waits up to the driver-timeout (~10 s), then fails: on JVM `Aeron.connect` throws a `DriverTimeoutException` (a subclass of `AeronException`); on the FFI path `clientConnect` returns NULL, which the generated binding raises as `FfiNullPointer`. The shared `AeronPlatform.external` primitive maps either, ONCE, to `Abort.fail(TopicTransportFailedException)`, so the failure is uniform across JVM, Native, JS, and Wasm and lands at the connect call site before any publish or stream runs. The mapping lives only in `external`: the JVM selector uses `Abort.catching[AeronException]`; the FFI selector uses `Abort.recover` over the `.safe.get` fiber bridge (the NULL surfaces as a panic through the `@Ffi.blocking` fiber, so it is recovered, not merely caught; any other failure or panic is re-raised as a defect). Both `Topic.run(aeronDir)` and `AeronClient.connect` consume this one primitive, so neither entry point can diverge into a panic, a null, or a backend-specific outcome.

Zero thread blocking is a module invariant, and the connect path honors it. The connect/driver-start downcalls are `@Ffi.blocking` (`AeronBindings.driverStart`, `AeronBindings.clientConnect`), so a slow connect does not freeze the JS event loop (the downcall runs on a libuv worker) or strand a Native/JVM carrier (it parks under the scheduler's blocking monitor); on JVM the synchronous `Aeron.connect` runs under `Sync.defer` and the scheduler detects the blocked carrier (thread-state and user-time monitoring) and keeps other fibers progressing on other workers, the same non-freeze effect. `TopicUniformInvariantsTest` "a concurrent ticker keeps ticking ..." pins this from both entry points: a concurrent ticker fiber keeps advancing (>= 50 ticks) throughout a ~10 s absent-driver connect, where a plain (non-`@Ffi.blocking`) binding would freeze the JS event loop to 0 ticks. See [No blocking, in any language](#no-blocking-in-any-language) for the discipline this is a case of.

The embedded driver runs the same uniform model with one isolation property: each `Topic.run(v)` instance allocates its own unique directory via `Path.tempDir` and passes it through `AeronPlatform.embedded(dir)` to `driverStart(dir)` / `clientConnect(dir)`, so two concurrent embedded drivers on one host never share Aeron's single default directory (where the second's `dirDeleteOnStart` would delete the first's CnC file). The directory is deleted in the `Sync.ensure` teardown after the driver is closed, leaving no leftover temp directories. This holds on all four backends; `TopicUniformInvariantsTest` "concurrent embedded Topic.run round-trips ..." pins the no-collision and no-leak behavior.

## No blocking, in any language

kyo-aeron never blocks a carrier thread, and the rule extends into the C shim.

- **No JVM blocking primitives in Scala main or test source:** no `Thread.sleep`, `synchronized`, `CountDownLatch`, `CyclicBarrier`, `LockSupport` / `parkNanos`, `Await.result` / `Await.ready`, `Thread.join`, or `Object.wait`. This is enforced by a grep gate over `kyo-aeron/{shared,jvm,native,js}/src` that must be empty before any commit. There is no sanctioned exception, including in tests.
- **No spin/yield primitives in the C shim:** no `usleep`, `nanosleep`, `sleep`, or `sched_yield` in `kyo_aeron.c`. The shim never busy-loops on registration: each `kyo_aeron_async_add_*_poll` is a single step that returns immediately, and the wait-and-retry loop lives in Scala `Async` (see the async add path below). The shim's only `for`/`while` is the capacity-doubling loop in the fragment handler, which terminates in bounded steps with no sleep or yield.

The Aeron C calls the shim wraps for the hot path (`offer`, `poll`, `is_connected`, the async-add steps) are non-blocking by contract, so those `AeronBindings` methods are plain synchronous downcalls. `driverStart` and `clientConnect` carry `@Ffi.blocking` because the driver-start and client-connect handshakes can block up to the driver timeout (~10 s on an absent-driver connect); `@Ffi.blocking` routes the downcall off the JS event loop (libuv worker) and parks the Native/JVM carrier under the scheduler's blocking monitor, never freezing the event loop or stranding a compute carrier. Backpressure and not-connected conditions are handled by a Scala-side bounded `Retry` loop in `Topic.scala` against `defaultRetrySchedule`, not by a blocked thread. The close downcalls (`driverClose`, `clientClose`) are plain synchronous calls that are not strictly non-blocking: `aeron_driver_close` joins the driver's conductor/sender/receiver pthreads. That join is bounded and runs only at `Topic.run` teardown, so it is left synchronous (on JVM/Native the scheduler's blocking monitor compensates; on JS it is a one-shot join at scope exit), unlike the unbounded ~10 s connect that `@Ffi.blocking` covers.

Tests wait deterministically with Kyo `Async` constructs: a publisher and its subscriber synchronize through a `Latch` so the subscriber is connected before the publish starts, and the test result is the subscriber fiber's `get`. `Latch.await` / `Fiber.get` are Kyo suspension, not thread blocking, and are allowed; the gate targets `java.util.concurrent` blocking and `Thread.sleep`. Never reach for a sleep to paper over a race; drive the ordering with a latch.

## The C shim memory model

`kyo_aeron.c` owns native memory and native threads. A contributor editing it must preserve the structures below: the context-lifetime pairing, the close-race locking, the async add split, the non-exiting error handler, and the grow-on-demand receive slot.

### Context lifetime

Each driver and each client is created from a context (`aeron_driver_context_t*` / `aeron_context_t*`) that must outlive the handle. The shim bundles the handle with its context and does not free the context at create time. `kyo_aeron_driver_close` calls `aeron_driver_close` then `aeron_driver_context_close`; `kyo_aeron_client_close` mirrors it with `aeron_close` then `aeron_context_close`. Close the handle before its paired context. A subscription bundle similarly pairs the fragment assembler and reassembly slot with the subscription; `kyo_aeron_subscription_close` frees them, or, when the free is deferred for the poll-after-own-close case (see the use-after-own-close invariant below), the client-close sweep does.

### Four close races, two locks, one lock order

The shim guards four distinct races on the client bundle. Two locks cover them: `close_mutex` (per bundle) and `g_registry_mutex` (process-global).

- **close-vs-close** (`close_mutex` + a `closing` flag). When `kyo_aeron_client_close` runs, the Aeron conductor's on-close path frees every still-registered publication and subscription struct. A later `kyo_aeron_publication_close` / `kyo_aeron_subscription_close` (from a `Sync.ensure` cleanup that outlives the `Topic.run` scope) would then dereference a freed struct. The close path sets `closing = 1` under `close_mutex`; the publication/subscription close checks `closing` under the same mutex and skips the raw Aeron close when the client is already gone.
- **add-vs-close** (`g_registry_mutex` + a process-global live-bundle registry). `kyo_aeron_async_add_publication` / `kyo_aeron_async_add_subscription` receive a raw client pointer whose bundle may already have been freed. They acquire `g_registry_mutex`, verify the bundle is still in the live list and not closing, and increment its refcount, all under the global lock, before dereferencing it. If the bundle is gone or closing, they return NULL, which the FFI bindings surface as `Absent` and `Topic` maps to `TopicPublicationClosedException`, never a segfault. After releasing `g_registry_mutex` they re-check `closing` under `close_mutex` (a TOCTOU guard: a racing close could free the inner `aeron_t` between the registry check and the `c->client` use) before the non-blocking `aeron_async_add_*` start call. `client_bundle_release` decrements the refcount under the global lock and frees the bundle (and removes it from the registry) only at zero.
- **hot-path-deref-vs-close** (`close_mutex` + the `closing` flag, extended to the inner-handle derefs). `kyo_aeron_publication_offer`, `kyo_aeron_publication_is_connected`, `kyo_aeron_publication_max_message_length`, and `kyo_aeron_subscription_poll` dereference the inner Aeron publication/subscription handle, which a concurrent `kyo_aeron_client_close` (calling `aeron_close`) can free out from under them. Each wraps only the `closing` read and the one non-blocking Aeron call under `close_mutex`: if `closing` is observed, it returns the existing safe sentinel (not-connected, `0` fragments, `AERON_PUBLICATION_CLOSED`) without touching the freed handle. The publication/subscription bundle's refcount keeps the client bundle (and therefore its `close_mutex`) alive for the duration of the call.
- **use-after-own-close invariant** (`close_mutex` + a per-bundle `closed` flag + per-client free-deferral, applied identically to publications AND subscriptions). Distinct from hot-path-deref-vs-close: here it is the bundle's OWN `kyo_aeron_publication_close` / `kyo_aeron_subscription_close` that has run, not a concurrent client close. A `Sync.ensure` cleanup can outlive the `Topic.run` scope, so a stray `offer` / `is_connected` / `max_message_length` (publication) or `poll` / `is_connected` (subscription) may legitimately run AFTER the caller's own close. Without the deferral, the close would free the bundle and that later call would dereference freed memory (a use-after-free on Native/JS). So `*_close` sets `b->closed = 1` under `close_mutex` and DEFERS the bundle free; the hot-path reads check `b->closed` (alongside `client->closing`) under `close_mutex` and return the EXISTING safe sentinel (publication `offer` -> `-4`, `is_connected` -> not-connected, `max_message_length` -> `0`; subscription `poll` -> `0` fragments, `is_connected` -> not-connected) without touching the dead Aeron handle. The publication `-4` is not a new sentinel: it maps through `AeronSentinels.Closed` to `TopicPublicationClosedException`, matching the JVM, where a closed refcounted `io.aeron.Publication.offer` returns `Publication.CLOSED` (-4) memory-safely; the subscription `0` maps to `Absent`. `*_close` is also idempotent: a second close on an already-`closed` bundle is a no-op, so a double close never double-frees the bundle nor double-releases the client refcount. `TopicUniformInvariantsTest` "offer after the caller's own closePublication ..." and "poll after the caller's own closeSubscription ..." pin the safe sentinels cross-platform.
  - **Free-ownership of the deferred bundle is exactly one owner.** Each publication bundle is on its client's intrusive `pub_list_head` and each subscription bundle on `sub_list_head` (both guarded by `close_mutex`). If the caller closes the bundle while the client is still open (`closing == 0`), the bundle stays in its list with `closed == 1` and the client-close sweep is its sole free-owner (it frees every list entry whose `closed == 1`, after `aeron_close` has already freed the inner handles, so the sweep frees only the bundle struct, plus the assembler + slot for a subscription). If the client is already closing when the bundle closes (`closing == 1`), the sweep already ran and skipped this bundle (its `closed` was still `0` then), so `*_close` itself unlinks and frees it. Exactly one free per bundle, no double-free; `TopicUniformInvariantsTest` "client-close after a caller-closed publication is clean ..." and "client-close after a caller-closed subscription is clean ..." pin it (clean teardown, `recordedFatalError` stays `Absent`).

The lock order is **`g_registry_mutex` before `close_mutex`**, always. Every site that needs both acquires the registry mutex first; no site acquires `close_mutex` then the registry mutex (the per-client publication-list helpers and the free-deferral sweep hold ONLY `close_mutex`, never the registry mutex, so they cannot invert it). Neither mutex is held across a slow Aeron close call: the critical sections wrap only a flag read plus one non-blocking Aeron call, never `aeron_close`. Preserve this ordering, the NULL-on-closed contract, the `closing`-guard on every inner-handle deref and the publication/subscription `closed`-guard, and the single-free-owner split when editing any `add_*`, `*_close`, or hot-path function.

### The async add path is a start/poll/get/free split

Registering a publication or subscription is not a single C call. The shim exposes a four-function split, symmetric for publications and subscriptions, so the wait happens in Scala `Async` and never blocks a carrier thread:

- `kyo_aeron_async_add_publication` (start) acquires the registry guard, kicks off `aeron_async_add_publication`, and returns an opaque heap token immediately, or NULL when the client bundle is closed/closing.
- `kyo_aeron_async_add_publication_poll` (one step) calls `aeron_async_add_publication_poll` once and returns `> 0` (done), `0` (awaiting), or `< 0` (the driver rejected the registration). On the `> 0` return Aeron writes the completed publication pointer into the out-param and frees the async handle, so the shim captures that pointer into the token at the `> 0` site; the handle must never be polled again. On the `< 0` return the shim captures `aeron_errcode()` / `aeron_errmsg()` into the token before any further Aeron call can overwrite the thread-local error.
- `kyo_aeron_async_add_publication_get` (retrieve) wraps the captured publication pointer in a bundle and frees the token. It does not re-poll.
- `kyo_aeron_async_add_publication_free` (cleanup) releases the client refcount and frees the token, called on a `< 0` poll or when the Scala fiber is interrupted mid-poll.

The shared `Topic.addPublicationDeadline` / `addSubscriptionDeadline` helpers drive this with a `Loop` that polls, sleeps `Async.sleep(addBackoff)` between steps, and is bounded by a `Clock` deadline; on expiry they abort with `TopicAddTimeoutException`. A `< 0` poll carrying a non-zero error code or non-empty detail aborts with `TopicRegistrationFailedException`; a `< 0` poll with `(0, "")` (the JVM closed-client `AeronException`) maps to `Absent`. A `Sync.ensure` finalizer calls the `_free` function if the fiber is cancelled while awaiting, so a token never leaks. No C busy-loop remains, no carrier thread is monopolized, and the loop is interruptible throughout. The `_err_code` / `_err_msg` token accessors are read after a `< 0` poll and before `_free`; the JVM impl mirrors the contract by extracting `RegistrationException.errorCodeValue()` / `.getMessage()`.

### Non-exiting error handler

The Aeron C client installs a default error handler that calls `exit(EXIT_FAILURE)` on a fatal conductor condition (driver timeout, conductor service timeout, command-buffer-full, broadcast error). On Native and JS that kills the process before the Scala layer can react. The shim installs its own non-exiting handler with `aeron_context_set_error_handler` before `aeron_init` (before the conductor thread that may fire it exists). The handler records the first fatal error (errcode plus message) into a mutex-guarded `kyo_aeron_error_slot` and returns; it never calls `exit`, `abort`, sleeps, or spins.

The slot is allocated before the handler is installed, stashed in the client bundle once the bundle is malloc'd, and freed in `client_bundle_release` only after `aeron_close` has joined the conductor thread (so the single writer is gone before the slot is freed). `kyo_aeron_has_client_error` / `kyo_aeron_client_error_msg` / `kyo_aeron_client_error_code` read the slot under the same mutex; `FfiAeronTransport.recordedFatalError` polls them at the offer/poll boundary and surfaces a present error as `TopicTransportFailedException`, deriving a non-empty detail from the errcode when the recorded message is empty (Aeron passes a NULL message for some conditions). The JVM mirrors this with `Aeron.Context.errorHandler` writing an `AtomicReference` slot. A test-only `kyo_aeron_test_inject_error` seam (and the matching `injectError` transport op) fires the same recording path with a synthetic error; it is always compiled in and inert in production.

### Grow-on-demand receive slot

The subscription owns a single reassembly slot that starts at 64 KiB (`KYO_AERON_SLOT_INITIAL_CAP`) and grows on demand toward the 16 MiB ceiling (`KYO_AERON_SLOT_MAX_CAP`, the Aeron protocol-level `AERON_FRAME_MAX_MESSAGE_LENGTH`), matching the JVM `FragmentAssembler`'s grow-on-demand profile. The C fragment handler is a `static` function fired synchronously by `aeron_subscription_poll` on the calling Scala carrier thread; it copies the reassembled message into the slot and never calls back into Scala. `kyo_aeron_subscription_poll` returns the byte count when the message fits the caller's `dst`, `0` for no message, `-1` for a poll error, `-2` for a message over the ceiling, and the negated required length when the message is reassembled but `dst` was too small (the message is retained, `pending = 1`, and copied out on the next poll without consuming a new Aeron message, so nothing is lost while the Scala-side reusable buffer grows). The slot, `slot_len`, `slot_cap`, and `pending` are accessed without synchronization under a strict one-poller-per-subscription contract, which the Scala side upholds by giving each `Topic.stream` subscriber its own subscription handle. Do not poll one subscription from two carriers.

### The fragment handler stays in C

The Aeron fragment-handler callback is a C function pointer the shim owns. Scala never registers a callback that Aeron could fire from an Aeron-owned thread. The embedded driver's conductor, sender, and receiver run on dedicated C pthreads (`AERON_THREADING_MODE_DEDICATED`, the post-init default; the shim does not call `set_threading_mode`) that never touch the Scala heap and never up-call into Scala. This is what makes the native-thread model safe on Scala Native (the GC ignores C-only threads) and on JS (no koffi foreign-thread upcall). Keep all callbacks in C.

## Build and link

Set the standard JVM options before building (see the root guide). Building auto-formats; re-read edited files afterward.

```sh
# JVM (links the shim object, uses the io.aeron Java client at runtime)
sbt 'kyo-aeron/test'
sbt 'kyo-aeron/testOnly kyo.TopicTest'

# Native and JS (link the C shim against staged libaeron)
sbt 'kyo-aeronNative/test'
sbt 'kyo-aeronJS/test'
```

### Staging libaeron

`kyo-aeron/scripts/build-aeron.sh <os-arch>` clones Aeron at the pinned 1.50.2 tag, builds the static archives with CMake, and stages them under `kyo-aeron/build/aeron/staged/<os-arch>/` (e.g. `darwin-aarch64`, `linux-x86_64`, `linux-aarch64`). The staged tree is gitignored: the archives are build artifacts, never committed binary blobs, so they must be produced on every machine and every CI runner before the compile step. The CI "Prepare libaeron" step runs `kyo-aeron/scripts/build-aeron.sh` per arch on the JVM, Native, JS, and Wasm legs (cmake is preinstalled or installed on demand). Run the script locally once per os-arch before a Native, JS, or Wasm build.

### The link is driver-only

The FFI library links **only** `aeron_driver_static`:

```scala
linkLibs = Seq("aeron_driver_static")
```

The driver archive already embeds the full client source (the driver CMakeLists inlines the client), so linking both `aeron_static` and `aeron_driver_static` produces hundreds of duplicate-symbol errors on Darwin ld64. The driver archive alone provides the complete client and driver API. Do not add `aeron_static` to `linkLibs`.

### Native and JS build wiring

The Native build passes the staged include dirs through `nativeConfig.compileOptions` so the shim's `#if __has_include(<aeronc.h>)` guard is true and the symbols are emitted; without the `-I` flags the guard is false and every function compiles out to an empty object, surfacing later as undefined-symbol link errors. The header-absent translation unit is a defensive no-op (a single unused typedef) rather than silently empty, so a misconfigured include path fails loudly. `AeronBindings` is configured `nativeBundled = true`: the shim is statically folded into the Native binary, so the codegen must not emit a `@link("kyo_aeron")` looking for a dynamic library that does not exist. The JS build installs the koffi range matching `FfiErrors.KoffiSupportedRange` and points the Node env at the compiled `libkyo_aeron-<os>-<arch>` shared object. Native tests run with `parallelExecution := false`. Each embedded driver uses a unique
per-instance directory from `Path.tempDir`, so concurrent drivers never share a CnC file.
Tests run serially because the UDP round-trip and URI-validation suites use fixed high
ports (e.g. 24519) that would collide across suites run concurrently, and because the embedded
driver and its test resources favor serial execution to avoid resource contention.

### After any AeronBindings edit, run a full clean

The kyo-ffi codegen caches generated `*Impl.scala` keyed on stale TASTy. `ffiClean` alone does not clear it, so a newly added binding method can be silently dropped and surface later as a Native or JS compile gap. After editing the `AeronBindings` trait, run a full `sbt clean` before re-validating on Native and JS. Adding a method to the trait also forces every test mock that implements it to gain a stub; those mock edits are a required consequence of the change, not scope drift.

## Cross-platform test discipline

Tests are cross-platform by default: a leaf is written `"name" in { ... }` (the kyo-test V3 DSL) and asserts the same concrete values on JVM, Native, JS, and Wasm. Every behavioral suite lives in `shared/src/test` and runs on all four platforms through the module `Test` base (`abstract class Test extends kyo.test.Test[Any]`, in `shared/src/test`). The one platform-folder test is `JvmAeronTransportTest` (`jvm/src/test`), which drives the io.aeron-specific `JvmAeronTransport` catch paths directly and has no cross-platform analogue. `TopicTest` asserts concrete `aeron:ipc` round-trip values; `TopicRoundTripTest` / `TopicValidationTest` assert the `aeron:udp` contract cross-platform too, using FIXED high ports (e.g. `127.0.0.1:24519`) under `Test / parallelExecution := false` rather than a `java.net.DatagramSocket` free-port helper (a JVM-only API that breaks the JS/Native/Wasm link). Assert concrete values, not just types or non-emptiness.

Source-shape and C-shim introspection are NOT tested with char-index / grep leaves (that asserts source shape, not behavior). These properties are enforced structurally instead: no `io.aeron.*` in `shared` by the build (io.aeron is a JVM-only dependency, so any `shared/` import fails the JS/Native compile); the C symbols match the bindings by the static link (a missing symbol fails `nativeLink` / `ffiCompile`); the fragment handler stays C-private at the C source site.

**A platform-specific leaf needs a bar-meeting MECHANIC rationale: a genuine platform mechanic with no cross-platform equivalent, named in a one-line comment at the leaf.** Platform gating uses the V3 filters `.onlyJvm` / `.notJvm` / `.onlyNative` / `.onlyJs`. "It's easier to write on one platform" does not meet the bar. The current legitimate platform-gated leaves are:

- **`TopicInvariantsTest` "runtime: symbols resolve and driver starts without system libaeron" (`.onlyNative`):** the Native static-link mechanic. Pins that the `kyo_aeron_*` symbols are statically folded into the Native binary (`aeron_driver_static`) and resolve at runtime with no system libaeron. The JVM has no C shim; JS dlopens via koffi (a different mechanism). The same file keeps "JS koffi int64 sentinel marshalling ..." (`.onlyJs`) for the koffi BigInt -> Long sentinel sign-marshalling, which only koffi's BigInt path exercises.
- **`AeronTransportTest` "UAF-reads: ... return safe sentinels, not a UAF" (`.notJvm`, i.e. Native + JS):** the C-shim freed-handle UAF guard. The `close_mutex` + `closing` per-function guard exists only in `kyo_aeron.c`, which Native and JS both link (koffi loads the same shim on JS); the JVM uses the memory-safe refcounted Java client and has no `kyo_aeron_*` symbols to guard. The leaf drives the raw bindings directly, so it must run wherever the C shim exists.
- **`AeronTransportTest` "oversize offer on term-length=65536 returns -6, does not throw" (`.onlyJvm`):** the JVM `IllegalArgumentException`-suppression mechanic. Only `io.aeron.Publication.offer` throws an IAE from `checkMaxMessageLength` on an oversize payload, which `JvmAeronTransport.offer` catches and normalizes to `-6`. The FFI path returns `-6` natively (no throw), so on Native/JS this leaf would pass for the wrong reason; the cross-platform `-6` contract is verified by the sibling "oversize offer on term-length=65536 returns -6 on every platform" leaf, which runs on all four platforms.

Every other leaf, including all of `TopicUniformInvariantsTest` (embedded-driver isolation, the uniform connect-failure and non-freeze leaves, both offer-after-own-close leaves, the errorCode-normalization leaves), is cross-platform `"name" in { ... }`.

## Pre-submission checklist (kyo-aeron)

Beyond the root checklist:

- [ ] All wire-contract logic (stream id, `Envelope`, codec, sentinel mapping) lives only in shared `Topic.scala` / `AeronSentinels`, never in a transport impl.
- [ ] `Topic`'s public effect rows match these rows:
  `run(v)` and `run(client)`: `A < (Async & S)`;
  `run(aeronDir)`: `A < (Async & Abort[TopicTransportFailedException] & S)`;
  `AeronClient.connect`: `AeronClient < (Scope & Async & Abort[TopicTransportFailedException])`;
  `publish`: `Unit < (Topic & S & Abort[TopicBackpressureException | TopicPublishException | TopicTransportException] & Async)`;
  `stream`: `Stream[A, Topic & Abort[TopicBackpressureException | TopicTransportException] & Async]`.
- [ ] No `io.aeron.*` or `org.agrona.*` token in `shared/` or on the public surface; those types stay in `jvm/src/main`.

- [ ] In `kyo_aeron.c`: handle closed before paired context; `g_registry_mutex` acquired before `close_mutex`; `async_add_*` returns NULL on a closed/closing bundle; every inner-handle deref runs under `close_mutex` with a `closing` check, returning the existing safe sentinel without dereferencing the dead handle, and the publication derefs (`offer`, `is_connected`, `max_message_length`) and subscription derefs (`poll`, `is_connected`) additionally check the bundle's `closed` flag for the use-after-own-close case; both `*_close` functions are idempotent and DEFER the bundle free, with exactly one free-owner (the client-close sweep when `closing == 0` at the caller's own close, else `*_close` itself); the non-exiting error handler is installed before `aeron_init` and its slot freed only after `aeron_close`; the reassembly slot grows on demand and the fragment handler stays `static` with no Scala upcall.
