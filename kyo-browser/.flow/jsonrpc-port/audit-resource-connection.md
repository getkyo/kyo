# kyo-browser Resource-Closing + Test-Cascade Audit

## Q1: `SharedChrome` lifecycle

`SharedChrome` caches exactly one piece of state: a `Promise.Unsafe[String, Abort[BrowserSetupException]]` named `cachedUrl` that holds the WebSocket debug URL of the shared Chrome process (e.g. `ws://127.0.0.1:<port>/devtools/browser/<uuid>`). It also carries an `AtomicBoolean.Unsafe` named `initStarted` used as a one-shot CAS gate. Both are static `val`s initialized before any Kyo handler is on the stack.

Shutdown happens exclusively on JVM/Node/Native scheduler exit. The `ensureStarted` path (SharedChrome.scala:53-74) forks a `Fiber.initUnscoped` that holds a `Scope.run` open with `Async.never`. When the kyo scheduler shuts down, the fiber is interrupted, the scope's finalizers run, and Chrome + its temp directory are torn down. There is no explicit `close`, no reference count, no `withBrowser`-level increment/decrement. Each `withBrowser` call (BrowserTest.scala:116) calls `SharedChrome.init` to get the URL and then calls `Browser.run(url)(...)`, which creates a fresh `CdpBackend` and a fresh browser context for that call. So the shared state in `SharedChrome` is only the URL string; the CDP connection and the browser context are per-call. If a test corrupts Chrome's page state (broken JS, crashed renderer), `SharedChrome` has no detection or recycling mechanism: it never re-checks Chrome liveness after the initial Q-002 probe. A crashed Chrome process would cause subsequent `CdpBackend.init` calls to fail at the WS probe with `BrowserSetupFailedException`, which `withBrowser` surfaces as an Abort rather than re-launching Chrome.

## Q2: `CdpBackend.close` / `closeNow`

The `close(gracePeriod)` sequence at CdpBackend.scala:107-110 is a sequential three-step chain: (1) `endpoint.close(gracePeriod)` — delegates to `JsonRpcEndpoint` which drains in-flight calls up to the grace period, then closes the WebSocket transport; (2) `Abort.run[Closed](dialogQueue.close).unit` — closes the dialog channel so the drainer fiber sees a `Closed` on its next `stream().foreach` iteration; (3) `dialogDrainer.interrupt.unit` — interrupts the drainer fiber explicitly. `closeNow` (CdpBackend.scala:113-116) is identical but passes no grace period to the endpoint. The dialog drainer fiber (built by `buildDialogDrainer` at CdpBackend.scala:560-585) runs `dialogQueue.stream().foreach(...)` inside `Abort.run[Closed]`, so the `Closed` from step 2 unblocks it cleanly; the explicit interrupt in step 3 is a belt-and-suspenders for the case where the fiber is between iterations. The ordering is important: if the endpoint closes first, any in-flight `Page.handleJavaScriptDialog` sends via `endpoint.sendUnmatched` in the drainer will fail with `Closed` (caught by the inner `Abort.run[Closed]` at CdpBackend.scala:574), which is safe.

The race between `close` and a concurrent in-flight `send` is handled by `JsonRpcEndpoint`: the endpoint's close path signals the in-flight pool so any call awaiting a response gets a synthetic `JsonRpcError` with message `"endpoint closed"`, which `CdpBackend.send` maps to `BrowserConnectionLostException` (CdpBackend.scala:64-68). There is no deadlock path because the endpoint's close does not wait for in-flight promises to be delivered; it fails them and then closes the transport.

## Q3: Per-test cleanup

Each `withBrowser` call (BrowserTest.scala:116) calls `Browser.run(url)(...)`. `Browser.run(wsUrl)` (Browser.scala:276-279) wraps its entire body in `Scope.run` and allocates a fresh `CdpBackend` via `CdpBackend.init(wsUrl, ...)`. Inside `CdpBackend.init` (CdpBackend.scala:153-226) every dispatcher table is freshly allocated: `dialogHandlers`, `dialogQueue`, `frameEventDispatchers`, `downloadEventDispatchers`, `dialogRecorders`, `lastEvaluateParams` are all `AtomicRef.init`/`Channel.initUnscoped` calls that create new objects for that specific backend. The `dialogDrainer` fiber is also freshly spawned per `initUnscoped` call (CdpBackend.scala:197). So every test calling `withBrowser` gets an entirely fresh `CdpBackend` with zero cross-test shared dispatcher state.

The browser context is also fresh per test: `BrowserTabSetup.attachAndSetupTab` (BrowserTab.scala:214-237) calls `CdpBackend.createBrowserContext`, registers `CdpBackend.disposeBrowserContext` via `Scope.ensure`, and creates a new target inside that context. Context disposal tears down cookies, localStorage, sessionStorage, service workers, and downloads for that context on scope exit. The `BrowserTab` itself allocates fresh `AtomicRef`s for `frameContexts`, `rootFrameId`, `consoleCaptureRegistered`, `responseTrackerRegistered`, `viewportOverride`, and `downloadPolicy` (BrowserTab.scala:58-76). Mock-fetch installations and cookies are isolated to the browser context and cleaned up on context disposal. The dialog drainer fiber is per-`CdpBackend`, so it is fresh per test. The `lastEvaluateParams` `AtomicRef` starts `Absent` per backend.

## Q4: Cascade-failure hypothesis rankings

**(a) Chrome process zombie — MEDIUM.** `SharedChrome` does not detect Chrome death and does not recycle. If Chrome crashes mid-test, the `cachedUrl` Promise still holds the stale URL. The next test's `CdpBackend.init` will try to WS-connect to a dead process, fail at the Q-002 probe (BrowserSetupFailedException), and surface that as an Abort that `withBrowser` propagates. This means subsequent tests fail with a setup error rather than a cascade of confusing mid-test failures. The failure mode is a clean Abort, not a silent corruption. Relevant file: SharedChrome.scala:36-37 (`cachedUrl.safe.get` returns whatever landed first — including a panic if Chrome died during startup, but not mid-run).

**(b) WebSocket port collision — LOW.** Chrome uses `--remote-debugging-port=0`, so the OS assigns a free port. The WebSocket URL includes a unique `/devtools/browser/<uuid>` path. Even if Chrome is restarted and reuses the port number, the UUID in the path differs, so a stale cached URL would fail the WS handshake rather than silently connect to the wrong session. The URL is cached once and never updated, so a port-number reuse cannot produce invisible cross-test contamination.

**(c) Shared `frameContexts` stale executionContextIds — LOW.** `frameContexts` is per-`BrowserTab` (BrowserTab.scala:24, allocated fresh at BrowserTab.scala:59). Each `withBrowser` call creates a new `BrowserTab` with an empty `Dict.empty` map. The `Scope.ensure` in `installFrameContextTracker` (BrowserTab.scala:94) removes the handler from `frameEventDispatchers` on scope exit, so the handler cannot outlive its tab. There is no cross-test sharing.

**(d) Phase 03 `dispatchDownloadEvent` broadcast — HIGH.** When a download event arrives with no session-specific handler or with an unmatched session, `dispatchDownloadEvent` (CdpBackend.scala:694-701) broadcasts to ALL entries in the `downloadEventDispatchers` table by folding over the entire map. The `downloadEventDispatchers` table is per-`CdpBackend`, and each test's `withBrowser` creates a fresh `CdpBackend`. So cross-test contamination of this table cannot occur. However, within a single test that installs a handler via `onDownload` and then fails (causing `close` to be called while an in-flight download event is being dispatched), the broadcast could invoke the test's handler on events that were already in-flight. The handler is cleaned up via `Scope.ensure` inside `onDownload`'s internal `Scope.run` (Browser.scala:2543-2553), which fires on Abort. The risk is confined to the duration of a single test's scope, not across tests.

**(e) Dialog drainer fiber from prior test — LOW.** The dialog drainer is per-`CdpBackend` (CdpBackend.scala:197). Each `withBrowser` call allocates a new `CdpBackend` and therefore a new drainer fiber. The `close` sequence interrupts the drainer (CdpBackend.scala:110). Even if the interrupt is slightly delayed, the drainer's `dialogQueue` is already closed (step 2 of `close`), so the drainer's `stream().foreach` unblocks via `Closed` and exits cleanly. No cross-test sharing.

**(f) `lastEvaluateParams` diagnostic — LOW.** `lastEvaluateParams` is an `AtomicRef[Maybe[String]]` per-`CdpBackend`. It holds the JSON params of the last `Runtime.evaluate` call for diagnostic purposes. It is written on every `runtimeEvaluate` (CdpBackend.scala:37-39) and never read by any action path. It cannot cause behavioral cascade failures, only confusing diagnostics.

**(g) `endpoint.close(grace)` hung mid-shutdown deadlock — HIGH.** The `close` sequence is: (1) `endpoint.close(gracePeriod)`, then (2) close dialog queue, then (3) interrupt drainer (CdpBackend.scala:107-110). If `endpoint.close(gracePeriod)` hangs because an in-flight call never receives a response (e.g. Chrome's CDP handler crashed, WS peer is silent but not closed), the entire `close` chain blocks for the full grace period. During this window, (2) and (3) have not run yet, so the drainer fiber is still alive and consuming from a non-closed channel. The `Scope.run` around the `CdpBackend.init` result does not complete until `close` returns (CdpBackend.scala:147: `Scope.acquireRelease(...)(_.close(launchCfg.closeGrace))`). If the outer ScalaTest test-runner has a timeout shorter than the grace period, the scope never completes cleanly and the underlying fiber resources (drainer, transport, Chrome connection) may leak into the next test's execution window. The grace period defaults to `LaunchConfig.default.closeGrace`; its value determines the maximum cascade window.

## Q5: Concrete fixes for BLOCKER findings

For **(g)** (endpoint close hung mid-shutdown): The `endpoint.close(gracePeriod)` call at CdpBackend.scala:108 should be wrapped in `Async.timeout(gracePeriod + 100.millis)(...)` to guarantee the entire `close` sequence completes within a bounded wall-clock time even if the endpoint's internal grace-period accounting stalls. Alternatively, run steps (2) and (3) (queue close + drainer interrupt) concurrently with `endpoint.close` via `Async.zip` so the drainer is always interrupted regardless of endpoint close duration. File: `kyo-browser/shared/src/main/scala/kyo/internal/CdpBackend.scala` line 107-110.

For **(d)** (broadcast hits stale-session handler): no fix needed for the cross-test scenario because the `downloadEventDispatchers` table is per-`CdpBackend` and is freshly allocated per test. If in-test leakage is observed (handler fires after the test's `Scope.run` completes), confirm that the `restoreDispatcher` `Scope.ensure` in `Browser.onDownload` (Browser.scala:2538-2541) fires before the drainer fiber is interrupted. The current order is `Scope.ensure(restoreDispatcher).andThen(Scope.ensure(channel.close))` which should unregister the dispatcher before any in-flight event can reach it. No change required unless the ensure ordering is inverted at a call site.
